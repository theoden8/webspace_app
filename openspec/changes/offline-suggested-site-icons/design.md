## Context

The icon stack is a module of top-level functions over module-level mutable caches, not a class: `getSvgContent`, `fetchIconBytes`, `getFaviconUrlStream` in `lib/services/icon_service.dart`, backed by `_faviconCache`, `_faviconQualityCache`, `_svgContentCache` (lines 59-63). Every network seam routes through one injectable factory, `outboundHttp.clientFor()` (`icon_service.dart` `_proxiedClient` line 26; `third_party/favicon/favicon.dart:92`). There is no bare `http.get`/`HttpClient` in the path, so "did this touch the network?" reduces to "was `outboundHttp.clientFor()` called?".

The suggestions grid (`lib/screens/add_site.dart` ~750-810) renders each tile via `UnifiedFaviconImage`, which drives `getFaviconUrlStream` ÔÇö i.e. rendering the list currently fetches from DDG/Google. `suggested_sites_service.dart:40-41` returns an empty default for fdroid precisely to avoid that contact; fmain ships the curated `kDefaultSuggestions` (21 sites).

Constraint that frames everything: the parallel goal is F-Droid reproducible builds. Any icon data shipped for the curated set must be a **committed, pinned asset**, never fetched during `flutter build` ÔÇö a build-time fetch is non-deterministic and just relocates the third-party contact to the build server.

## Goals / Non-Goals

**Goals:**
- Rendering the suggestions list makes zero outbound icon requests (provable, not best-effort).
- Curated default suggestions show real icons offline, sourced from committed bundled assets.
- Adding a suggestion or custom site still fetches and stores its icon (cache warming).
- fmain and fdroid ship identical suggestions defaults.
- A regression test fails if the list ever reaches the network.

**Non-Goals:**
- Changing proxy routing, `OutboundClientReady`/`OutboundClientBlocked` semantics, or the outbound factory.
- Bundling icons for user-added custom suggestions (those warm at add-time).
- Offline behavior anywhere outside the suggestions list (main site grid, nested webviews keep fetching).
- A class refactor of the icon module (rejected below).

## Decisions

### D1. Offline read = a dedicated read path, not a class/singleton refactor
Add an offline entry point alongside the existing functions (e.g. `Stream<String> suggestionIconStream(String url)` / `String? cachedFaviconUrl(String url)`) that consults bundled assets + `_faviconCache`/`_svgContentCache` + persisted disk cache, and **never calls `_proxiedClient`**. On miss it resolves to a placeholder and completes; it does not fetch.

- Alternative considered: refactor the module into `IconService.instance` (network) + `IconService.offline` (a `final` offline field). Rejected: the module is functional with global caches; rerouting every caller (`add_site.dart`, `main.dart`, `UnifiedFaviconImage`) buys only "hold a reference instead of call a function." Not worth the blast radius.
- Alternative considered: a mutable global `offline` bool toggled around the list screen. Rejected: ambient mutable state + the UI re-entrancy class CLAUDE.md calls out (set/clear races across concurrent consumers, `onPopInvokedWithResult` re-entry).

### D2. One shared cache, read by both paths
The offline path reads the same module-level caches the network path writes. So when the add action fetches an icon, a subsequent offline list read sees it in-session with no restart. This is automatic given D1 (no forked caches). Requirement-level invariant goes in the `icon-fetching` spec.

### D3. Curated icons are committed assets, regenerated by a script
Ship `assets/suggested_icons/<key>.<ext>` in the APK, registered in `pubspec.yaml`. A developer-run `scripts/` regenerator fetches current favicons for `kDefaultSuggestions` and writes the asset files; it is run by hand and the output committed, never invoked by `flutter build`. Asset key derives deterministically from the suggestion (host-based) so the offline lookup can map a suggestion URL to its bundled asset without a manifest.

- Alternative considered: fetch-at-build via a gradle/dart build step. Rejected: breaks reproducibility and moves DDG/Google contact to the build server (the exact thing we are removing).
- Alternative considered: embed icons as base64 in Dart source. Rejected: bloats source, worse diffs, no real benefit over an asset.

### D4. Two data sources feed one offline read
Curated suggestions Ôćĺ bundled assets. User-added custom suggestions Ôćĺ persisted cache warmed at add-time. Both surface through the same offline read path; the path tries bundled asset first, then cache, then placeholder.

### D5. Reconcile flavors only after the guarantee holds
`flavorDefaultSuggestions` stops branching on fdroid (returns `kDefaultSuggestions` for all flavors). This ships in the same change as the offline path and bundled assets, never before ÔÇö shipping it first would reintroduce the privacy regression on fdroid. `test/suggested_sites_test.dart` flavor expectation updates with it.

### D6. Offline is distinct from blocked
Do not express offline by feeding the suggestions context an `OutboundClientBlocked` factory. `OutboundClientBlocked` means "proxy could not be honored, fail closed for privacy" ÔÇö a security seam. Keep "told not to fetch" and "cannot fetch safely" as separate concepts; the offline path simply never calls the factory.

## Risks / Trade-offs

- [Bundled icons go stale vs the live favicon] Ôćĺ Acceptable: suggestion thumbnails are recognizers, not authoritative; the regenerator script refreshes them on demand and the add-time fetch still gets the live icon when the user actually adds the site.
- [A curated suggestion has no bundled asset (regen missed it)] Ôćĺ The offline path shows a placeholder; no network leak. A test over `kDefaultSuggestions` asserts every entry has a committed asset so this surfaces in CI, not at runtime.
- [Asset-key collision across suggestions sharing a host] Ôćĺ Key derivation must be unique per suggestion; covered by a test over `kDefaultSuggestions` keys.
- [Reproducibility: committed binary assets in the diff] Ôćĺ They are pinned inputs (like vendored upstream), not build outputs; the regenerator is the runner, the assets are the committed source. Consistent with CLAUDE.md "don't commit derivatives" because they cannot be regenerated deterministically (a favicon today != next month), so they are pinned, not generated at build.
- [APK size grows with N committed icons] Ôćĺ Small (favicons are KB-scale); bounded by `kDefaultSuggestions` size.

## Migration Plan

1. Land offline read path + bundled assets + regen script (no behavior change yet; list still uses network path).
2. Switch the suggestions grid tiles to the offline read path; add the zero-network test.
3. Flip `flavorDefaultSuggestions` to drop the fdroid branch; update flavor test.
4. Rollback: revert step 3 (fdroid empty default) independently of 1-2 if a leak is found; the offline path and assets are inert without the flavor flip.

## Open Questions

- Asset format: keep favicons in their fetched format (ico/png/svg mixed) or normalize to png at regen time? Normalizing simplifies the offline loader's content-type handling.
- Does the offline list need SVG color-detection parity, or is a raw bundled asset render sufficient for thumbnails (skip the `_isSvgColored` quality path entirely offline)?
- Placeholder on miss: reuse the existing letter/initial fallback widget, or a neutral globe glyph?
